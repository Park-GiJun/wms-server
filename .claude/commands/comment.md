# /comment — 주석 보강

대상 코드에 이 저장소의 주석 컨벤션에 맞는 **한국어 주석을 추가/개선**한다.
**코드 로직은 한 글자도 바꾸지 않는다** — 주석만 만진다.

## 입력

- 대상: **`$ARGUMENTS`** (파일/디렉터리 경로, 여러 개 가능)
  - 생략 시 `git status --porcelain` 의 변경·신규 파일 중 소스(`.kt`/`.sql`/`.yml`)만 대상.
  - 변경 파일도 없으면 사용자에게 대상을 물어본다.

## 주석 컨벤션 (이 저장소 스타일 — 기존 코드가 기준)

`AppUserModel`, `ItemCommandHandler`, `V2__app_user.sql` 등 기존 파일의 주석 밀도·톤을 따른다.

1. **한국어**로 쓴다. 기술 용어·식별자·API 경로는 원문 유지 (예: "append-only 원장", "`X-User-Id` 헤더").
2. **코드가 스스로 말하지 못하는 것만** 쓴다:
   - 도메인 규칙·불변식 (예: "PENDING 에서만 승인/거절 가능", "음수재고 차단")
   - 설계 의도·트레이드오프 (예: "계정 존재 여부를 숨기려 자격 검증이 먼저")
   - 경계·책임 (예: "엔티티↔모델 변환만 담당 — 로직 없음")
   - HTTP 매핑 결과 (예: "코드 중복 시 409")
3. **금지**: 라인이 뭘 하는지 반복하는 주석("id 를 반환한다"), 변경 이력·작성자,
   "이 수정이 왜 옳은지" 리뷰어용 설명. 이런 주석이 이미 있으면 삭제 제안(직접 삭제는 주석에 한함).
4. 배치 기준:
   - **클래스/파일 상단 KDoc**: 역할 한 줄 + 지키는 불변식/경계 (public 클래스는 원칙적으로 필수)
   - **함수 KDoc**: 이름만으로 알 수 없는 규칙이 있을 때만 (예: `/** login — ACTIVE 만 발급. */`)
   - **인라인 `//`**: 특정 라인의 비자명한 제약만 (예: `// 72 = BCrypt 입력 상한`)
   - **SQL**: 테이블 상단에 용도·수명주기 한 줄, 비자명한 컬럼·인덱스에 이유
5. 이미 충분한 주석이 있는 코드는 **건드리지 않는다**. 고치는 것보다 안 고치는 게 기본값.

## 절차

1. 대상 파일을 확정하고 전부 Read 한다.
2. 파일별로 위 컨벤션 기준 "주석이 필요한데 없는 곳"과 "금지 유형 주석"을 찾는다.
   판단이 서지 않는 곳은 주석을 달지 않는다(과잉 주석이 누락보다 나쁘다).
3. Edit 으로 주석만 추가/수정한다. KDoc 은 `/** … */`, 참조는 `[대상]` 링크 문법을 쓴다.
4. 보고: 파일별로 `추가/수정한 주석 위치 · 한 줄 내용` 목록. 로직 변경이 없음을 명시한다.

## 규칙

- 주석 외의 diff 가 생기면 안 된다 (import·공백·포매팅 포함). 커밋은 하지 않는다 — `/commit` 은 별도.
- 대상에 테스트 파일이 있으면 테스트 이름이 이미 설명이므로 클래스 상단 한 줄 외에는 달지 않는다.
- 생성 파일(빌드 산출물, gradle wrapper 등)은 제외한다.
