# 비밀번호 정책을 domain service 로 분리 + 단위 테스트

- **일시**: 2026-07-05
- **커밋**: feat[user-service] : 비밀번호 정책을 domain service 로 분리 + 단위 테스트

## 작업 내용
web DTO 의 `@Size` 어노테이션에만 있던 비밀번호 규칙을 순수 도메인 서비스 `PasswordPolicy` 로 분리했다(업무 규칙의 원본을 domain 레이어로). register 핸들러가 해싱 전에 검증하며, 위반 시 `WeakPasswordException`(400, 사유 포함)을 던진다. user-service 의 첫 단위 테스트(`PasswordPolicyTest`, 6케이스)를 추가했고 빌드·테스트 통과.

## 변경 파일
- `backend/user-service/src/main/kotlin/.../domain/appUser/service/PasswordPolicy.kt` (신규) — 8~72자(BCrypt 상한)·영문·숫자 포함·공백 금지 규칙, 상태/의존성 없는 object
- `backend/user-service/src/main/kotlin/.../domain/appUser/exception/AppUserException.kt` — `WeakPasswordException`(INVALID_INPUT/400) 추가
- `backend/user-service/src/main/kotlin/.../application/handler/command/AppUserCommandHandler.kt` — register 에서 `PasswordPolicy.validate` 호출
- `backend/user-service/src/test/kotlin/.../domain/appUser/service/PasswordPolicyTest.kt` (신규) — 통과/길이/영문/숫자/공백 6케이스

## 주요 결정사항
- domain service 분리 기준: "한 엔티티에 못 담기는 순수 업무 규칙"만 — 상태 전이는 모델에 유지(anemic model 회피), 중복검증·로그인 흐름은 application 에 유지.
- web 의 `@Size` 는 1차 방어로 존치, 정책 원본은 `PasswordPolicy` 단일 파일.
