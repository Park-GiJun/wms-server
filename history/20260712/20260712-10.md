# 이동 확정 시 stock.movement Kafka 발행

- **일시**: 2026-07-12 19:39
- **커밋**: feat[master-service] : 이동 확정 시 stock.movement Kafka 발행

## 작업 내용
재고원장 5단계(마지막 단계)로, 커맨드가 확정한 재고 이동을 stock.movement Kafka 토픽으로 발행하는 기능을 master-service 에 추가했다. 발행 포트/어댑터를 신설해 트랜잭션 커밋 이후에만 이벤트가 나가도록 보장하고, StockCommandHandler 가 receive/adjust/transfer 각 커맨드에 맞게 이벤트를 매핑·발행하도록 연결했다. 이로써 커맨드→원장→발행으로 이어지는 파이프라인이 완성됐다.

## 변경 파일
- `backend/master-service/src/main/kotlin/.../application/port/out/message/PublishStockMovementPort.kt` — 신규. 발행 포트, "커밋 이후에만 발행" 계약을 KDoc 에 명시
- `backend/master-service/src/main/kotlin/.../infrastructure/adapter/out/message/StockMovementKafkaAdapter.kt` — 신규. 트랜잭션 활성 시 TransactionSynchronization.afterCommit 으로 발행 예약(롤백된 이동은 발행 안 됨), key="{skuId}:{locationId}" 로 파티션 내 순서보장, 토픽은 inventory.topic.stock-movement 프로퍼티 주입, 발행 실패는 로그만(원장은 이미 확정)
- `backend/master-service/src/main/kotlin/.../application/handler/command/StockCommandHandler.kt` — 포트 주입, receive/adjust 1건·transfer 2건(출발지 먼저) 발행, toEvent() 매퍼(movementId = 원장 PK 문자열 = 컨슈머 멱등 키)
- `backend/master-service/src/test/kotlin/.../StockCommandHandlerTest.kt` — FakePublishPort 추가, transfer 발행 순서(출발지→도착지 2건) 테스트 추가(총 8케이스)
- `config-repo/master-service.yml` — kafka.producer 에 String/Json serializer 추가. inventory.allow-negative 가정값 키 삭제(조정만 음수 예외 허용이라는 주석이 실제 구현과 모순되는 죽은 설정), itemId 주석을 skuId 로 정정
- `config-repo/application.yml` — kafka 주석의 itemId → skuId 정정

## 주요 결정사항
1. AFTER_COMMIT 책임을 어댑터에 둠 — 핸들러는 트랜잭션 안에서 publish() 호출만, "언제 보내는가"는 기술 관심사라 infrastructure 소관.
2. "커밋됐는데 발행 전 크래시" 유실 갭은 알려진 트레이드오프 — 문제가 되면 outbox 패턴으로 승격(후속).
3. movementId = 원장 PK — persist() 가 id 채워진 movement 를 반환하는 이유.
4. 토픽명은 하드코딩 대신 기존에 예약돼 있던 inventory.topic.stock-movement 설정 주입.
