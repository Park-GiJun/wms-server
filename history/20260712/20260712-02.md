## 작업 내용
상품/SKU 마스터(직전 커밋)에 이어 재고원장 (sku, location) 축의 나머지 한쪽인 로케이션 마스터를 구현했다. 상품 슬라이스와 동일한 헥사고날 + CQRS 컨벤션으로 domain/application/infrastructure 전 레이어를 작성했고, Flyway V4 마이그레이션(location 테이블) 포함. gateway 라우트는 /api/locations/** 가 master-core 에 이미 있어 변경 불필요. 도메인 단위 테스트(LocationModelTest, 4케이스) 추가, 전체 빌드·테스트 통과(BUILD SUCCESSFUL).

REST API: POST /api/locations(생성, 코드 중복 409), GET /api/locations?status=&zone=(목록, 필터 조합), GET /api/locations/{id}, PUT /api/locations/{id}(zone/type 만 수정), PATCH /api/locations/{id}/status(활성/비활성).

## 주요 결정사항
1. 로케이션 계층(창고>존>셀)은 zone 문자열 필드로 단순 표현 — 필요해지면 트리 구조로 승격.
2. LocationStatus 는 ItemStatus 와 값(ACTIVE/INACTIVE)이 같지만 도메인별 enum 으로 분리 — 도메인 간 교차 결합 방지.
3. code 는 비즈니스 키로 불변(수정 API 는 zone/type 용도 변경만). 마스터는 삭제 대신 status=INACTIVE 전환, 같은 상태 전환은 409.
4. 목록 필터는 status/zone 두 축 — 어댑터에서 derived query 4조합(둘 다/각각/전체)으로 처리.
5. gateway 라우트 변경 없음 — /api/locations/** 는 master-core 라우트에 이미 존재.
