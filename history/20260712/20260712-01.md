- **일시**: 2026-07-12 14:39

## 작업 내용

재고원장(척추) 구현에 앞서 그 전제인 상품 마스터를 구현했다. 상품(Product)과 SKU 를 1:N 으로
분리 — 상품은 카탈로그/집계 단위이고 재고의 주인은 항상 SKU 다(재고원장은 skuId 만 참조). 기존
appUser 슬라이스와 동일한 헥사고날 + CQRS 컨벤션으로 domain/application/infrastructure 전
레이어를 작성했고, Flyway V3 마이그레이션(product/sku/sku_option 테이블)과 gateway 라우트
갱신까지 포함했다. 도메인 단위 테스트(ItemModelTest, 6케이스)를 추가했으며 전체 빌드·테스트
통과를 확인했다(BUILD SUCCESSFUL).

REST API: `POST/GET /api/products`, `GET /api/products/{id}`(SKU 포함 상세),
`PUT /api/products/{id}`, `PATCH /api/products/{id}/status`, `POST /api/products/{id}/skus`,
`GET /api/skus?skuCode=|barcode=`(바코드 스캔 대응 단건 조회), `PUT /api/skus/{skuId}`,
`PATCH /api/skus/{skuId}/status`.

## 주요 결정사항

1. Product(1):SKU(N) 분리 — 재고원장은 skuId 만 참조하고 상품은 원장에 등장하지 않는다. 상품별 총재고 같은 집계는 조회(QueryHandler) 관심사.
2. 옵션 없는 상품도 SKU 1개 강제 생성(상품 생성 커맨드에 SKU ≥1 필수) — 원장이 아는 ID 를 skuId 하나로 유지.
3. 마스터는 삭제 대신 status=INACTIVE 전환(soft delete) — 원장이 참조하는 마스터라 하드 삭제 금지. 같은 상태로의 전환은 409 로 거부.
4. code/skuCode 는 비즈니스 키로 불변(수정 API 에서 제외). barcode 는 nullable + UNIQUE(스캔 대응 GET /api/skus?barcode= 제공).
5. JPA 에서 product↔sku 연관관계 매핑을 하지 않고 productId 값만 참조(어그리게이트 경계 분리). 옵션은 @ElementCollection(sku_option 테이블, 수정 시 전체 교체).
6. UOM/포장 단위(BOX=12EA)는 이번 범위에서 제외 — 원장 qty 는 항상 최소 단위 고정, 필요 시 환산 테이블로 추가.
7. 상품 변경 API 에 역할(ADMIN) 제한은 미적용(게이트웨이 인증만) — 마스터 관리 권한 정책 확정 시 X-User-Role 체크 추가.
