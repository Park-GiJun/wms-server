## 작업 내용
상품/SKU·로케이션(직전 두 커밋)에 이어 마스터 3종의 마지막인 거래처 마스터를 구현했다. 입출고 피처가 참조할 최소 골격(code/name/type/status)만 담았고, 연락처·주소 등은 실제 필요 시 확장한다. 동일한 헥사고날 + CQRS 컨벤션으로 전 레이어 작성, Flyway V5 마이그레이션(partner 테이블) 포함. gateway 라우트는 `/api/partners/**` 가 master-core 에 이미 있어 변경 불필요. 도메인 단위 테스트(PartnerModelTest, 4케이스) 추가, 전체 빌드·테스트 통과(BUILD SUCCESSFUL). 이로써 마스터 3종(상품·로케이션·거래처) 완료 — 남은 큰 덩어리는 재고원장 코어.

REST API: `POST /api/partners`(생성, 코드 중복 409), `GET /api/partners?status=&type=`(목록, 필터 조합), `GET /api/partners/{id}`, `PUT /api/partners/{id}`(name/type 만 수정), `PATCH /api/partners/{id}/status`(활성/비활성).

## 주요 결정사항
1. 최소 골격 원칙 — code/name/type/status 만. 연락처·주소 등 상세 필드는 입출고 피처가 실제 필요로 할 때 확장.
2. PartnerType 에 BOTH 추가 — 공급도 하고 구매도 하는 양방향 거래처를 두 레코드로 쪼개지 않기 위함.
3. code 는 비즈니스 키로 불변(수정 API 는 name/type 만). 마스터는 삭제 대신 status=INACTIVE 전환, 같은 상태 전환은 409.
4. 목록 필터는 status/type 두 축 — 입고 화면은 type=SUPPLIER, 출고 화면은 type=CUSTOMER 로 조회하는 용도. 어댑터에서 derived query 4조합 처리.
5. gateway 라우트 변경 없음 — `/api/partners/**` 는 master-core 라우트에 이미 존재.
