# StockMovementEvent 계약을 재고원장 설계에 맞게 정비

- **일시**: 2026-07-12 17:19
- **커밋**: refactor[shared] : StockMovementEvent 계약을 재고원장 설계에 맞게 정비

## 작업 내용
wms-server 의 척추 이벤트 StockMovementEvent(shared 모듈)는 아직 발행처가 0곳이라 지금이 계약을 바꿀 마지막 무료 기회였다. 재고원장(StockMovement append-only + StockBalance (skuId, locationId) 단일 라이터) 구현 착수 전에 이벤트 계약을 확정했다. 전체 빌드(gradlew build) 통과.

## 변경 파일
- `backend/shared/src/main/kotlin/com/gijun/wms/shared/event/StockMovementEvent.kt`
  - `locationId: String` → `Long` — 로케이션 마스터 PK(LocationJpaEntity.id: Long)와 타입 일치. 원장 잔고 키 (skuId, locationId) 가 둘 다 BIGINT 로 정렬됨
  - `refType`/`refId` → nullable — ADJUSTMENT 처럼 유발 문서가 없는 이동이 있음
  - `seq` 주석 정정 — "전역 증가"가 아니라 (skuId, locationId) 쌍 안에서 단조 증가, 컨슈머의 순서/멱등 판단 기준임을 명시
  - `qty` 주석 정정 — 부호 있는 수량(+유입/-유출, 0 금지). TRANSFER 는 같은 타입으로 ±2건이라 type 만으로 부호가 정해지지 않음
  - KDoc 의 itemId 잔재 제거, Kafka key 표기를 "{skuId}:{locationId}" 로 수정, TRANSFER=2건 분해 규칙(출발지 -qty + 도착지 +qty, 같은 refType/refId 공유, 그래서 from/to 필드 없음)을 계약 문서에 명시
  - data class 뒤의 빈 `{}` 제거

## 주요 결정사항
1. TRANSFER 를 이벤트 2건으로 분해 — (sku, location) 단위 단일 라이터 모델 유지의 핵심.
2. refType/refId 는 nullable String — 서비스 간 참조 유연성을 위해 문자열 유지.
3. 참조처가 shared 자신뿐이라 다른 모듈 수정 없음.
